# SwRS 编写

## 前置判断：CTS 完备性评估

**执行前必须先评估CTS的详细程度**，根据评估结果选择不同路径：

### 评估维度（3级判断）

| 级别 | 条件 | 路径 |
|------|------|------|
| **L3 - 完整** | CTS已到SWC级别 + 接口完整 | → 快速路径 |
| **L2 - 概要** | CTS有模块划分 + 接口待定义 | → 标准路径 |
| **L1 - 粗略** | CTS仅功能描述，无模块划分 | → 完整路径 |

### 评估输出格式

```
CTS完备性评估结论: [L3 / L2 / L1]
评估依据:
- [逐条列出评估依据]
选择路径: [快速路径 / 标准路径 / 完整路径]
```

## 路径1：L3 快速路径（智能转换模式）

**适用条件**：CTS已到SWC级别 + 接口完整

**场景特点**：
- 系统规范已包含完整的模块划分（SWC清单）
- 系统规范已包含详细的接口定义（信号表、服务表）
- 系统规范已包含执行逻辑（触发条件、执行动作）
- 主要工作是**格式转换+信息重组+补充完善**，而非从头推导

### 输入依赖

- **必须**：CTS文档（包含SWC清单 + 接口定义）
- **可选**：边界定义文档（已有则引用，无则从CTS中提取）

### 执行方式

1. 从CTS提取SWC模块列表和接口定义
2. 对每个模块的接口，生成对应的SwRS卡片
3. 补充缺失的例外处理（从触发条件反面推导）
4. 补充缺失的验证方法（从输入输出设计）

### 参数说明

```bash
--level=L3                # 指定为L3级别
--mode=smart-convert      # 使用智能转换模式
  - smart-convert：格式转换 + 信息重组 + 补充完善（L3推荐）
  - derive：从头推导需求（L1推荐）
```

### 智能转换的字段映射规则

对于系统功能规范表格格式，执行以下映射：

| 系统规范字段 | SwRS字段 | 转换逻辑 |
|------------|---------|---------|
| 需求映射 | 追溯来源 | 直接引用（如 CTS-001） |
| 软件模块 | 需求ID前缀 | 提取模块名（如 MFF） |
| 触发条件 | 触发条件 | 直接引用或简化 |
| 执行动作 | 预期行为 | 直接引用，提取关键行为 |
| 输入信息 | 输入 | 提取信号名、类型、范围 |
| 输出信息 | 输出 | 提取信号名、类型、值 |
| （缺失） | 例外处理 | **推导**：触发条件反面 |
| （缺失） | 验证方法 | **推导**：从输入输出设计 |
| （生成） | 需求ID | **生成**：SwRS-[模块]-[功能]-[序号] |

### SwRS卡片模板（L3 智能转换）

```markdown
| 字段 | 内容 |
|------|------|
| 需求ID | SwRS-[模块]-[功能]-[编号] |
| 功能名称 | [简洁功能名] |
| 触发条件 | [精确描述触发该功能的事件或状态] |
| 输入 | [(信号名, 数据类型, 取值范围/枚举)] |
| 预期行为 | [量化描述，如"在 X ms 内完成 Y"] |
| 输出 | [(信号名, 数据类型, 输出值/范围)] |
| 例外处理 | [异常输入或边界条件下的行为，由触发条件反面推导] |
| 追溯来源 | [CTS中的模块名或需求ID] |
```

## 路径2：L2 标准路径

**适用条件**：CTS有模块划分 + 接口待定义

### 输入依赖

- **必须**：CTS功能描述
- **必须**：边界定义文档（补充接口信息）

### 执行方式

1. 从CTS提取功能描述
2. 执行边界定义（补充接口信息）
3. 为每个模块推导SwRS
4. SwRS卡片中"追溯来源"字段填写模块名

### SwRS卡片模板（L2）

```markdown
| 字段 | 内容 |
|------|------|
| 需求ID | SwRS-[模块]-[功能]-[编号] |
| 功能名称 | [简洁功能名] |
| 触发条件 | [精确描述触发该功能的事件或状态] |
| 输入 | [(信号名, 数据类型, 取值范围/枚举)] |
| 预期行为 | [量化描述，如"在 X ms 内完成 Y"] |
| 输出 | [(信号名, 数据类型, 输出值/范围)] |
| 例外处理 | [异常输入或边界条件下的行为] |
| 追溯来源 | [模块名] |
```

## 路径3：L1 完整路径

**适用条件**：CTS仅功能描述，无模块划分

### 输入依赖

- **必须**：FC 功能清单（来自 fc-extract）
- **必须**：模块分解结果（来自 module-decomp）
- **必须**：边界定义文档

### 执行方式

1. 从FC功能清单提取功能
2. 执行模块分解
3. 执行边界定义
4. 为每个模块推导SwRS
5. SwRS卡片中"追溯来源"字段填写FC-ID

### SwRS卡片模板（L1）

```markdown
| 字段 | 内容 |
|------|------|
| 需求ID | SwRS-[模块]-[功能]-[编号] |
| 功能名称 | [简洁功能名] |
| 触发条件 | [精确描述触发该功能的事件或状态] |
| 输入 | [(信号名, 数据类型, 取值范围/枚举)] |
| 预期行为 | [量化描述，如"在 X ms 内完成 Y"] |
| 输出 | [(信号名, 数据类型, 输出值/范围)] |
| 例外处理 | [异常输入或边界条件下的行为] |
| 追溯来源 | [对应的 FC-ID] |
```

## 通用推导规则（所有路径）

1. SMART 原则：Specific, Measurable, Achievable, Relevant, Testable
2. 所有数值参数必须量化，不允许模糊词
3. 每条 SwRS 需说明可验证方式
4. 必须包括：输入、输出、功能逻辑

## 量化检查

| 模糊描述 | 量化要求 |
|---------|----------|
| "低速" | "< 30 km/h" |
| "快速响应" | "响应时间 ≤ 100 ms" |
| "合理范围" | "[最小值, 最大值]" |

## ID 命名示例

- `SwRS-MFF-HMISwitch-01`
- `SwRS-MFF-conf-01`
- `SwRS-MFF-UNP-01`

## 执行示例

```bash
# L3快速路径（CTS已包含完整SWC和接口，使用智能转换模式）
/swrs-write ./CTS文档.docx --level=L3 --mode=smart-convert

# L3快速路径（仅指定等级，使用默认的smart-convert模式）
/swrs-write ./CTS文档.docx --level=L3

# L2标准路径（CTS有模块，需要边界定义）
/swrs-write ./CTS文档.md ./边界定义.md --level=L2

# L1完整路径（完整流程）
/swrs-write ./FC清单.md ./模块分解.md --boundary=./边界定义.md --level=L1

# 通过 require skill 自动识别等级并执行
/require ./CTS文档.docx --path=fast
```
