# CTS 准入审查

作为软件端的"守门人"，确保不清晰的需求不转入软件分析阶段。

## 输入依赖

- **必须**：CTS 文档

## 检查项

按以下维度评估，给出 [✅/⚠️/❌] 状态：

- [必须] CTS 文档是否存在且包含可识别的功能描述？
- [推荐] 是否有通信矩阵（DBC/LDF/CSV/SOA）？
- [推荐] 是否能识别目标 SWC 或模块名称？
- [推荐] 是否有软件层划分说明（SWC清单/架构图）？
- [推荐] 是否明确了安全等级要求（ASIL）？

## 处理规则

对每个 ⚠️ 或 ❌ 项，给出：
1. 该缺失对后续步骤的影响范围
2. 可接受的替代方案或推断规则
3. 建议向用户补充哪些信息

## 输出格式

### 准入评审报告

```markdown
# CTS 准入评审报告

| 项目 | 状态 |
|------|------|
| 文档完整性 | ✅/⚠️/❌ |
| 功能逻辑闭环 | ✅/⚠️/❌ |
| 性能指标量化 | ✅/⚠️/❌ |
| 边界条件明确 | ✅/⚠️/❌ |
```

### 澄清清单

```markdown
## 需求澄清清单

| 序号 | 缺失项 | 对后续影响 | 替代方案 | 建议补充信息 |
|------|--------|------------|----------|--------------|
```

## 执行示例

```bash
/cts-intake ./CTS_泊车功能.docx
```
