# CTS 准入审查

作为软件端的"守门人"，确保不清晰的需求不转入软件分析阶段。

## 输入依赖

- **必须**：CTS 文档

## 检查项

按以下维度评估，给出 [✅/⚠️/❌] 状态：

- [必须] CTS 文档是否存在且包含可识别的功能描述？
- [推荐] 是否有通信矩阵（DBC/LDF/CSV/SOA）？
- [推荐] 是否能识别目标 SWC 或模块名称？
- [推荐] 是否有软件层划分说明（SWC清单/架构图）？
- [推荐] 是否明确了安全等级要求（ASIL）？

## 处理规则

对每个 ⚠️ 或 ❌ 项，给出：
1. 该缺失对后续步骤的影响范围
2. 可接受的替代方案或推断规则
3. 建议向用户补充哪些信息

## 完备性等级评估

**基于上述检查项，自动判定CTS完备性等级**：

| 级别 | 判定条件 | 推荐路径 |
|------|---------|---------|
| **L3** | ✅ 文档完整 + ✅ 软件模块划分 + ✅ 接口定义完整 | 快速路径（cts-intake → swrs-write → quality-gate） |
| **L2** | ✅ 文档完整 + ✅ 功能逻辑闭环 + ⚠️ 接口定义待补充 | 平衡路径（cts-intake → boundary-def → fc-extract → swrs-write → quality-gate） |
| **L1** | ⚠️/❌ 多项缺失，仅有功能描述 | 完整路径（执行全部6个步骤） |
